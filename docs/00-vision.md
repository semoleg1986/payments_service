# Архитектурное Видение Payments Service

## Назначение

`payments_service` — bounded context управления оплатами и доступом к курсам.
На текущем этапе сервис реализует ручной контур подтверждения оплаты без банковского эквайринга:
- родитель создает заявку на оплату
- администратор подтверждает оплату
- сервис фиксирует выдачу доступа к курсу

## Бизнес-ценность

Сервис обеспечивает:
- единый источник истины по оплатам и решениям по доступу
- прозрачный audit trail: кто создал/подтвердил/отклонил
- идемпотентную обработку заявок
- интеграцию с attribution-каналами и скидками

## Границы

### Входит в контекст
- `PaymentIntent`
- `PaymentDecision` (approve/reject)
- `CourseAccessGrant`
- `PaymentStatus`
- журнал событий оплаты и доступа

### Не входит в контекст
- выпуск JWT и identity lifecycle (`auth_service`)
- контент курсов и прогресс обучения (`course_service`)
- пользовательские профили (`users_service`)
- онлайн-эквайринг банка (следующий этап)

## Принципы

- Domain first, инварианты в домене
- Application facade как вход в use-cases
- Интеграции только через порты
- Контракты версионируются и тестируются
